# 项目规则

> AI 助手必须遵守的项目级规则。违反本规则的行为将导致代码质量下降或项目风险。

---

## 一、文件操作

### 1.1 安全删除

- **禁止**使用 `rm` 命令删除文件
- 必须使用 `mv` 命令将文件移至 `.trash/` 目录
- 示例：`mv 待删文件 .trash/`
- 定期清理 `.trash/` 中超过 7 天的文件

### 1.2 文件创建

- 创建新文件前，先确认目录是否存在
- 新文件必须有明确的用途注释（文件头注释）
- 禁止创建空文件（除非测试需要的占位文件）

### 1.3 敏感文件

- **绝对禁止**将密码、密钥、Token 提交到 Git
- 使用环境变量或 `.env` 文件（`.env` 必须加入 `.gitignore`）
- 示例 `.gitignore` 规则：
  ```
  .env
  *.key
  *.pem
  *.p12
  secrets/
  ```

## 二、代码编写

### 2.1 代码风格

- 严格遵循项目现有的代码风格（缩进、命名、导入顺序）
- 使用项目已配置的 lint 工具，提交前必须通过检查
- 新增代码必须有注释说明**为什么**（Why），不是**做什么**（What）

### 2.2 函数/方法规范

- 单个函数不超过 50 行
- 参数不超过 4 个（超过时使用对象/结构体封装）
- 函数职责单一，一个函数只做一件事
- 禁止返回 `null` / `None`（使用空对象、Optional 或抛出异常）

### 2.3 错误处理

- **禁止**空的 `try-catch`（吞掉异常）
- 所有外部依赖调用（网络、数据库、文件）必须有异常处理
- 错误信息必须包含上下文，方便排查
- 用户可见的错误信息不能暴露内部细节（堆栈、路径、SQL）

### 2.4 禁止项

- **禁止**使用 `eval()` / `exec()`（除非绝对必要且有安全审查）
- **禁止**硬编码路径、URL、IP 地址（使用配置）
- **禁止**全局变量（除非是只读常量）
- **禁止**注释掉的代码块（删除它，Git 有历史记录）
- **禁止** `console.log()` / `print()` 调试代码提交到主分支

## 三、测试

### 3.1 测试覆盖

- 新功能必须包含单元测试
- Bug 修复必须先写复现测试
- 核心业务逻辑覆盖率不低于 80%

### 3.2 测试质量

- 测试名称描述**预期行为**（`should_return_empty_when_user_not_found`）
- 每个测试只验证一个场景
- 使用真实数据，避免 Mock 一切（Mock 只用于外部依赖）
- 测试必须能独立运行（不依赖执行顺序）

### 3.3 测试执行

- 提交前必须运行全部测试
- 测试失败时**禁止**直接跳过或标记 skip
- 先修代码，再修测试，不能反过来

## 四、Git 规范

### 4.1 分支策略

```
main          ← 稳定分支，禁止直接修改
├── feature/* ← 新功能
├── fix/*     ← Bug 修复
├── refactor/*← 重构
└── chore/*   ← 维护任务
```

### 4.2 提交信息

```
type: 简短描述

可选的详细描述（解释 WHY）

type 取值: feat, fix, refactor, chore, docs, test, style, perf
```

示例：
```
fix: 修复空邮箱地址未验证的问题

登录接口未校验邮箱为空的情况，导致空邮箱用户可以注册。
添加空值校验和错误返回。
```

### 4.3 提交粒度

- 每个提交只包含一个逻辑变更
- **禁止**混入无关修改（格式化、重命名、功能修改混在一起）
- 大功能分多次提交，每次可独立编译/测试

### 4.4 提交前检查

- [ ] 所有测试通过
- [ ] 无调试代码
- [ ] 无敏感信息
- [ ] 代码通过 lint
- [ ] 无未使用的导入/变量

## 五、安全

### 5.1 输入验证

- 所有用户输入必须校验（类型、长度、格式、范围）
- SQL 查询使用参数化语句，**禁止**拼接
- 文件上传验证类型、大小、内容

### 5.2 认证授权

- 每个接口必须有权限检查（不仅前端检查，后端也要）
- 密码必须哈希存储（bcrypt / Argon2），**禁止**明文或 MD5
- Token 必须有过期时间

### 5.3 输出安全

- 用户数据输出前必须转义（防 XSS）
- 错误响应不能包含堆栈信息、文件路径、SQL 语句
- API 响应统一格式，不暴露内部数据结构

### 5.4 依赖安全

- 不使用已知的漏洞依赖（定期检查 `npm audit` / `pip-audit`）
- 引入新依赖前检查其活跃度和安全性

## 六、性能

### 6.1 数据库

- **禁止** N+1 查询（使用 JOIN 或预加载）
- 频繁查询的字段必须有索引
- 大数据量操作使用分页或游标，**禁止**一次加载全部

### 6.2 内存

- **禁止**将大文件全部读入内存（使用流式处理）
- 循环中避免创建新对象（复用、对象池）
- 及时释放不再使用的资源（关闭连接、文件句柄）

### 6.3 网络

- 外部 API 调用设置超时（默认不超过 30 秒）
- 使用连接池，**禁止**每次请求新建连接
- 批量操作使用批量接口，**禁止**循环单个调用

## 七、文档

### 7.1 文档更新义务

**Agent 必须在以下时机主动更新项目文档：**

| 触发条件 | 更新文档 | 时机 |
|----------|----------|------|
| 收到新需求 | `.trae/rules/requirements.md` | 收到需求时立即 |
| 需求变更/完成 | `.trae/rules/requirements.md` | 变更/完成时 |
| 做出技术决策 | `.trae/rules/project_memory.md` | 决策完成后 |
| 发现关键信息 | `.trae/rules/project_memory.md` | 发现后立即 |
| 遇到并解决错误 | `.trae/rules/lessons_learned.md` | 解决后 |
| 踩坑总结 | `.trae/rules/lessons_learned.md` | 总结后 |
| 学到新技能 | `.trae/skills/skills_summary.md` | 学会后 |

### 7.2 README 维护

- 项目结构变化时更新目录结构说明
- 新增功能时更新使用说明
- 环境变量/配置变更时更新配置文档

## 八、AI 开发红线（绝对不能违反）

### 8.1 不确定的事情

- **禁止**猜测 API 地址、端口、数据库连接信息
- **禁止**假设外部服务一定可用（必须有超时和重试）
- **禁止**假设文件一定存在（必须先检查）
- 不确定时**必须问用户**，不能瞎猜

### 8.2 不熟悉的代码

- **禁止**在不理解的代码上直接修改
- 先阅读、理解、再修改
- 修改前先说明修改思路，得到确认后再动手

### 8.3 不验证就提交

- **禁止**不运行测试就声称"修复完成"
- **禁止**不手动/自动验证就声称"功能正常"
- 验证步骤必须在回复中展示执行结果

### 8.4 过度工程

- **禁止**在不需要时引入设计模式、抽象层、工厂模式
- **禁止**为"未来可能需要"的功能写代码（YAGNI）
- 能用简单方案解决时，不要复杂方案

### 8.5 不完整实现

- **禁止**写 TODO 作为功能未完成的借口（TODO 只用于已知的、有计划的改进）
- 开始实现就必须完成
- 不能完成的必须在开始前说明原因

## 九、多语言项目规则

### 9.1 Python

- 使用 type hints
- 遵循 PEP 8
- 使用 `black` / `ruff` 格式化
- `requirements.txt` 或 `pyproject.toml` 锁定依赖版本

### 9.2 JavaScript/TypeScript

- 使用 TypeScript 优先（除非项目明确使用 JS）
- 启用 `strict` 模式
- 使用 ESLint + Prettier
- `package.json` 使用精确版本号（`1.2.3` 而不是 `^1.2.3`）

### 9.3 Go

- 遵循 `go fmt` 格式
- 使用 `go mod tidy` 管理依赖
- 错误处理**禁止** `_ =` 忽略错误
- 公开函数/变量必须有大写首字母 + 文档注释

### 9.4 Java

- 遵循项目约定的命名规范
- 使用 Lombok 减少样板代码（如果项目使用）
- 使用 `@Nullable` / `@NonNull` 标注空安全
- Maven/Gradle 锁定依赖版本

## 十、部署规范

### 10.1 环境分离

- 开发、测试、生产环境配置完全分离
- 环境变量管理，**禁止**硬编码环境差异
- 生产环境开启只读文件系统（如适用）

### 10.2 部署检查清单

- [ ] 所有测试通过
- [ ] 无已知安全漏洞的依赖
- [ ] 数据库迁移脚本已测试
- [ ] 环境变量已配置
- [ ] 日志级别正确（生产不为 DEBUG）
- [ ] 备份已创建（如涉及数据变更）
- [ ] 回滚方案已准备

### 10.3 回滚

- 每次部署必须有回滚方案
- 数据库变更必须向后兼容（支持旧代码运行）
- 灰度发布优先于全量发布
